1. üéØ Purpose

This protocol outlines how CareAI manages feature flags to control the rollout of functionality across clinical, caregiver, and user environments. It ensures:

  • Safe testing and phased deployment of AI/ML and personalization features
  • Configurable user experiences by role, consent, and platform type
  • Full reversibility and traceability for any adaptive behavior

3. üß© Flag Architecture

Feature flags are stored in a remote config schema accessible by:

  • Companion logic engine
  • Frontend (tablet, mobile)
  • Backend services

Flags are:

  • Version-controlled
  • Environment-scoped (dev, staging, production)
  • Logged on evaluation

6. üìã Metadata Requirements

Every flag must include:

  • Unique ID (feature_group.flag_name)
  • Creation date
  • Author/owner
  • Target roles and environments
  • Default behavior if flag fails (e.g., off or fallback mode)

8. üßØ Emergency Disable Procedure

If a flagged feature malfunctions:

  1. Admin uses admin control panel or CLI tool to disable the flag.
  1. Event is logged with override reason and actor.
  1. Users fall back to prior behavior immediately (no app restart needed).
